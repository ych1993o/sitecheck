HTTP安全头配置指南：CSP、HSTS、X-Frame-Options

2026年6月11日 · SiteCheck

在 Web 安全防御中，HTTP 安全头（Security Headers）是一道常被忽视却极其重要的防线。它们通过一行行响应头配置，就能有效防御 XSS、点击劫持、MIME 嗅探、协议降级等多种攻击。与代码层面的防护相比，安全头配置简单、生效迅速、成本低廉，是性价比极高的安全加固手段。本文将详解几个关键安全头的作用与配置方法，帮助你构筑网站的安全防线。响应头的基础知识可先阅读HTTP 响应头详解。

一、HTTP 安全头的作用与价值

HTTP 安全头是服务器通过响应头向浏览器下达的"安全指令"，告诉浏览器在处理页面时遵循哪些安全策略。例如：只允许加载同源脚本、强制使用 HTTPS、禁止被嵌入 iframe。这些指令在浏览器层面执行，能在攻击发生前就阻断威胁。

安全头的价值在于"纵深防御"：即使应用代码有漏洞，安全头也能提供一层额外保护。比如 CSP 能在 XSS 漏洞被利用时阻止恶意脚本执行。配置安全头几乎不影响性能，却能显著提升安全评级。用 securityheaders.com 等工具扫描，能直观看到安全头的完备程度。

二、Strict-Transport-Security (HSTS)

HSTS（HTTP Strict Transport Security）强制浏览器始终通过 HTTPS 访问网站，防止 SSL 剥离和降级攻击。没有 HSTS，用户即使输入 HTTPS 网址，也可能被中间人降级到 HTTP。

配置示例：

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

参数说明：

  • max-age=31536000：HSTS 有效期 1 年（秒），期间浏览器强制 HTTPS。
  • includeSubDomains：策略覆盖所有子域名。确认所有子域名都支持 HTTPS 后再加。
  • preload：申请加入浏览器内置的 HSTS 预加载列表，即使用户首次访问也强制 HTTPS。

注意事项：HSTS 一旦生效，在 max-age 内无法撤销（浏览器会持续强制 HTTPS）。建议先用短 max-age（如 300 秒）测试，确认全站 HTTPS 稳定后再设为一年。加入 preload 列表需谨慎，因为移除缓慢。这是迁移 HTTPS后的重要加固步骤。

三、Content-Security-Policy (CSP)

CSP（内容安全策略）是防御 XSS 最强大的安全头，通过白名单机制限制页面能加载哪些资源、能执行哪些脚本。合理配置 CSP 能让即使存在 XSS 注入点，恶意脚本也无法执行。

配置示例：

Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' https://api.example.com; font-src 'self' https://fonts.gstatic.com; frame-ancestors 'none'; base-uri 'self'; object-src 'none'

常用指令说明：

  • default-src 'self'：默认只允许同源资源，是 CSP 的兜底策略。
  • script-src：限制 JS 来源，是防 XSS 的核心。
  • style-src：限制 CSS 来源。'unsafe-inline' 允许内联样式，会降低安全性。
  • img-src：限制图片来源。data: 允许 base64 图片。
  • connect-src：限制 AJAX/Fetch/WebSocket 的连接目标。
  • frame-ancestors 'none'：禁止页面被任何站点 iframe 嵌入，替代 X-Frame-Options。
  • object-src 'none'：禁止 Flash、Java 等插件。

CSP 配置建议用"先报告后 enforcing"的方式逐步收紧：

  1. 先用 Content-Security-Policy-Report-Only 头，只报告违规不拦截，收集日志。
  2. 分析报告，把合法资源加入白名单。
  3. 报告稳定无遗漏后，切换为正式的 Content-Security-Policy 强制执行。
  4. 持续监控报告，发现新增资源及时调整策略。

CSP 配置复杂，但安全收益极高，是值得投入的安全加固。

四、X-Frame-Options

X-Frame-Options 防御点击劫持（Clickjacking），控制页面是否允许被 iframe、frame、object 嵌入。攻击者可以把你的页面透明叠在诱导按钮上，骗用户点击触发敏感操作。

配置示例：

X-Frame-Options: DENY

取值：

  • DENY：完全禁止任何页面嵌入。
  • SAMEORIGIN：只允许同源页面嵌入。
  • ALLOW-FROM uri：已废弃，现代浏览器不支持。

X-Frame-Options 已被 CSP 的 frame-ancestors 取代，但为兼容旧浏览器，建议两者都配置。如果已配 CSP frame-ancestors，X-Frame-Options 可设 SAMEORIGIN 或 DENY 作为补充。

五、X-Content-Type-Options

设为 nosniff，禁止浏览器对资源类型进行 MIME 嗅探。没有这个头，浏览器可能把声明为 text/plain 的文件当脚本执行，引发安全隐患。

X-Content-Type-Options: nosniff

这是最简单却必备的安全头，一行配置就能避免 MIME 嗅探攻击，建议所有站点都启用。

六、Referrer-Policy

控制 Referer 头的发送策略，保护用户隐私，避免把完整 URL（含查询参数、token）泄露给第三方。

Referrer-Policy: strict-origin-when-cross-origin

常用取值：

  • no-referrer：完全不发送 Referer。
  • same-origin：同源请求才发送。
  • strict-origin-when-cross-origin（推荐）：同源发送完整 URL，跨源只发送 origin，HTTPS 到 HTTP 不发送。
  • no-referrer-when-downgrade：HTTPS 到 HTTPS 发送，HTTPS 到 HTTP 不发送。

strict-origin-when-cross-origin 是现代浏览器的默认值，兼顾隐私和功能，推荐显式声明。

七、Permissions-Policy

Permissions-Policy（原 Feature-Policy）控制页面能使用哪些浏览器 API 和功能，如摄像头、麦克风、地理位置、支付等。即使页面被注入恶意脚本，也无法调用被禁用的功能。

Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=()

() 表示完全禁用该功能，self 表示允许同源使用。按需配置，禁用站点不需要的功能，缩小攻击面。

八、完整配置示例

Nginx 配置

server {
    listen 443 ssl http2;
    server_name example.com;

    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
    add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; object-src 'none'; frame-ancestors 'none'; base-uri 'self'" always;
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;

    # 其余配置...
}

关键点：always 参数确保即使返回错误响应（如 404、500）也带上安全头；在 location 块中重复 add_header 会覆盖继承，需要把安全头在所有 location 都声明，或用 include 复用。

Apache 配置

<IfModule mod_headers.c>
    Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"
    Header always set Content-Security-Policy "default-src 'self'; script-src 'self'; object-src 'none'; frame-ancestors 'none'"
    Header always set X-Frame-Options "SAMEORIGIN"
    Header always set X-Content-Type-Options "nosniff"
    Header always set Referrer-Policy "strict-origin-when-cross-origin"
</IfModule>

九、配置后的检测与验证

配置完成后必须验证是否生效：

  • SiteCheck HTTP Headers 工具：用 HTTP Headers 检测查看响应头，确认安全头都已配置。
  • securityheaders.com：扫描网站安全头并给出评分（A+ 到 F），是评估安全头完备度的标准工具。
  • curl：curl -I https://example.com 快速查看响应头。
  • 浏览器开发者工具：Network 面板查看 Response Headers。
  • CSP 报告：查看 CSP Report-Only 的报告端点，确认没有误拦合法资源。

目标是通过 securityheaders.com 的扫描拿到 A 或 A+ 评级。

十、常见问题与注意事项

  • CSP 误伤功能：CSP 太严格会拦截合法资源导致功能异常。务必用 Report-Only 模式先测试，逐步收紧。
  • HSTS 误启用：全站没做好 HTTPS 就开 HSTS 会让 HTTP 用户彻底无法访问。先确保 HTTPS 稳定。
  • add_header 覆盖问题：Nginx 中 location 块的 add_header 会覆盖 server 块的，导致安全头丢失。用 include 复用或每个 location 重复声明。
  • 第三方资源：统计、广告等第三方脚本要在 CSP 白名单中放行，否则会被拦截。
  • 内联脚本/样式：CSP 默认禁止内联，需要改用 nonce 或 hash，或临时用 'unsafe-inline'（不推荐长期使用）。
  • CDN/反向代理：确认 CDN 或代理没有剥离或覆盖你配置的安全头。

十一、安全头配置优先级

如果一次性配置所有安全头有难度，建议按优先级分步实施：

  1. 第一优先：X-Content-Type-Options、X-Frame-Options——配置最简单，无副作用。
  2. 第二优先：HSTS、Referrer-Policy——HTTPS 站点必配。
  3. 第三优先：CSP——收益最大但配置最复杂，用 Report-Only 模式逐步推进。
  4. 第四优先：Permissions-Policy——按需禁用不需要的浏览器功能。

十二、总结

HTTP 安全头是性价比极高的安全加固手段，几行配置就能防御 XSS、点击劫持、降级攻击、MIME 嗅探等多种威胁。本文详解了 HSTS、CSP、X-Frame-Options、X-Content-Type-Options、Referrer-Policy、Permissions-Policy 等关键安全头的作用与配置，并给出 Nginx/Apache 示例。核心要点：HSTS 强制 HTTPS，CSP 是防 XSS 核心但需逐步收紧，X-Content-Type-Options 和 X-Frame-Options 简单必备，Referrer-Policy 保护隐私。

安全头配置不是一劳永逸，要随业务变化持续维护——新增第三方资源要更新 CSP 白名单，新功能上线要评估是否需要调整策略。建议你今天就用自己的网站用 securityheaders.com 扫描一次，对照本文补齐缺失的安全头，逐步把安全评级提到 A+。配合 SSL 证书（参考证书检测）和 HTTPS 迁移，你的网站安全防线将更加坚固。
