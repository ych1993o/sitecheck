DNS记录类型详解：A、CNAME、MX、TXT、SRV全解析

2026年6月8日 · SiteCheck

DNS 记录是域名系统的"数据单元"，每条记录定义了域名与某种资源（IP、别名、邮件服务器、文本等）的映射关系。配置域名时，面对 A、AAAA、CNAME、MX、TXT、SRV 等一堆记录类型，新手常感到困惑：它们各自做什么？什么时候该用哪种？本文将全面解析各类 DNS 记录类型的含义、格式与应用场景，帮你彻底搞懂 DNS 记录。如果你想先了解配置流程，可阅读DNS 配置教程。

一、DNS 记录的基本结构

每条 DNS 记录通常包含几个字段：名称（主机名/域）、类型（Type）、TTL（缓存时间）、值（记录内容）。在 DNS 区域文件中，一条记录看起来像这样：

www  IN  A     3600  192.0.2.1

这表示 www.example.com 的 A 记录指向 IP 192.0.2.1，TTL 为 3600 秒。在大多数 DNS 服务商的控制台里，你只需填写主机名、选择类型、填入值，TTL 用默认即可。

二、A 记录：域名到 IPv4

A 记录是最基础最常用的记录，把域名映射到一个 IPv4 地址。当你访问 www.example.com，A 记录告诉浏览器它的服务器 IP 是什么。

  • 名称：@（根域）、www、blog 等
  • 值：IPv4 地址，如 192.0.2.1
  • 用途：网站指向服务器 IP

一个名称可以配多条 A 记录指向不同 IP，实现 DNS 轮询负载均衡。一个名称也可以同时配 A 和 AAAA 记录，支持双栈。

三、AAAA 记录：域名到 IPv6

AAAA 记录（Quad-A）与 A 记录对应，把域名映射到 IPv6 地址。随着 IPv6 普及，配置 AAAA 记录越来越重要。

  • 名称：同 A 记录
  • 值：IPv6 地址，如 2001:db8::1
  • 用途：支持 IPv6 访问

支持双栈的客户端会优先尝试 IPv6（AAAA），失败再回退 IPv4（A）。配置 AAAA 前确保服务器 IPv6 真的可用，否则可能导致部分用户访问变慢（等待 IPv6 超时后才回退）。

四、CNAME 记录：域名别名

CNAME 记录把一个域名别名指向另一个域名（规范名），而不是直接指向 IP。常用于把子域名指向第三方服务。

  • 名称：如 cdn、blog
  • 值：目标域名，如 example.github.io
  • 用途：指向 CDN、托管服务、SaaS 平台

CNAME 的几条重要规则：

  • 根域名（@）通常不能配 CNAME，因为根域必须有 SOA 和 NS 记录，RFC 不允许与其他记录共存。部分现代 DNS 服务商通过 CNAME Flattening 绕过此限制。
  • 一个名称配了 CNAME 就不能配其他类型记录（如 A），CNAME 必须是该名称的唯一记录。
  • CNAME 指向的目标必须能正常解析，否则别名也解析失败。
  • CNAME 会有额外的解析步骤（先解析别名再解析目标），略微增加延迟。

五、MX 记录：邮件交换

MX 记录指定负责接收该域名邮件的邮件服务器。没有 MX 记录，邮件就无法投递到 @example.com。

  • 名称：通常 @
  • 值：优先级 + 邮件服务器域名，如 10 mail.example.com
  • 用途：配置企业邮箱、邮件服务

MX 记录的优先级数字越小越优先。发件方按优先级从大到小尝试（数字小优先），主服务器不可用时自动用备用服务器。注意 MX 的值必须是域名（有 A 或 AAAA 记录），不能是 IP。一个域名可配多条 MX 实现邮件高可用。

六、TXT 记录：文本记录

TXT 记录用于存储任意文本，如今主要用于域名验证和安全策略。虽然叫"文本"，但常用特定格式表达结构化信息。

主要用途：

  • SPF：声明哪些服务器有权以该域名发信，防邮件伪造。如 v=spf1 include:_spf.google.com ~all。
  • DKIM：存放公钥用于邮件签名验证，记录名形如 selector._domainkey。
  • DMARC：定义 SPF/DKIM 验证失败后的处理策略，记录名 _dmarc。如 v=DMARC1; p=quarantine; rua=mailto:dmarc@example.com。
  • 域名所有权验证：Google Search Console、云服务等要求添加特定 TXT 记录证明域名所有权。
  • 其他：site verification、BIMI、MTA-STS 等。

一个名称可配多条 TXT 记录，但单条 TXT 值有长度限制（通常 255 字符，超长需拆分）。

七、SRV 记录：服务定位

SRV 记录用于指定提供特定服务的服务器，比 MX 更通用——MX 专用于邮件，SRV 可用于任何服务。

SRV 记录的格式较复杂：优先级 权重 端口 目标。

_sip._tcp  IN  SRV  10  60  5060  sipserver.example.com

名称格式为 _服务._协议，如 _sip._tcp、_xmpp._tcp。

  • 优先级：同 MX，数字小优先。
  • 权重：同优先级内的负载比例。
  • 端口：服务监听端口。
  • 目标：提供服务的主机域名。

SRV 常用于 SIP（VoIP）、XMPP（即时通讯）、Active Directory、Kerberos 等服务的自动发现和定位。

八、NS 记录：权威服务器

NS 记录指定哪些域名服务器是该域名的权威服务器，即由谁来解析这个域名的记录。

  • 名称：@ 或子域名
  • 值：权威服务器域名，如 ns1.cloudflare.com
  • 用途：委托 DNS 解析权

当你把域名从 A 服务商迁到 B 服务商，本质就是修改注册商处的 NS 记录，指向 B 的权威服务器。NS 变更的传播时间最长（24-48 小时），因为涉及顶级域名服务器的缓存。NS 的值必须是域名而非 IP。

九、CAA 记录：证书授权限制

CAA（Certification Authority Authorization）记录指定哪些 CA 机构有权为该域名签发证书，防止证书被未授权的 CA 签发。

@  IN  CAA  0 issue "letsencrypt.org"
@  IN  CAA  0 issuewild ";"
@  IN  CAA  0 iodef "mailto:security@example.com"
  • issue：允许指定 CA 签发证书。
  • issuewild：控制通配符证书签发。";" 表示禁止。
  • iodef：违规签发 attempt 的报告地址。

配置 CAA 能降低证书被恶意签发的风险，是 HTTPS 站点的安全加固措施。更多证书知识见SSL 证书检测。

十、其他常见记录类型

  • SOA：起始授权机构记录，每个区域必须有一条，包含主服务器、管理员邮箱、序列号、刷新/重试/过期时间。一般由 DNS 服务商自动生成。
  • PTR：反向解析记录，把 IP 反查回域名，常用于邮件服务器反垃圾。配置在 IP 所属的 in-addr.arpa 区域。
  • ANAME / ALIAS：部分服务商提供的扩展记录，类似 CNAME 但可用于根域名（根域指向另一域名）。
  • HTTPS / SVCB：较新的记录类型，用于服务绑定和 HTTPS 记录提示，可加速 HTTPS 连接建立。

十二、如何查询 DNS 记录

配置或排查时，查询记录是必备操作：

  • SiteCheck DNS 工具：用 DNS 查询可视化查询所有记录类型，适合不熟悉命令行的用户。
  • dig：dig example.com A、dig example.com MX、dig example.com ANY，分别查不同类型。
  • nslookup：nslookup -type=mx example.com，跨平台。
  • host：host -t txt example.com，简洁输出。

查询时注意：不同 DNS 解析器的缓存可能不同，刚修改的记录可能要等 TTL 过期才查到新值。排查时可用 dig @8.8.8.8 指定解析器，或 +trace 追踪完整解析链路。

十三、DNS 记录配置常见问题

  • 根域名不能配 CNAME：用 A 记录或支持 CNAME Flattening 的服务商。
  • 邮件发不出：检查 MX 记录是否正确，SPF/DKIM/DMARC 是否配置。
  • 子域名解析失败：确认记录添加在正确的域名区域，主机名拼写无误。
  • CNAME 与 A 冲突：一个名称只能有一种，不能同时配 CNAME 和 A。
  • 记录修改不生效：可能是 TTL 缓存未过期，等待或调小 TTL。
  • CAA 阻止证书签发：CA 不在 CAA 白名单会拒绝签发，确认白名单包含你的 CA。

十四、总结

DNS 记录类型虽然多，但每种都有明确的用途。本文全面解析了 A、AAAA、CNAME、MX、TXT、SRV、NS、CAA 等核心记录类型的含义、格式与应用场景。核心记忆：A/AAAA 指向 IP，CNAME 指向别名，MX 配邮件，TXT 做验证和安全策略，SRV 定位服务，NS 委托解析权，CAA 限制证书签发。理解了每种记录的作用，配置域名时就能有的放矢。

DNS 是网站的基础设施，记录配置正确才能保证访问、邮件、证书等各项服务正常。建议把本文作为速查参考，配置时对照格式规范操作，并用查询工具验证。结合配置教程和传播原理，你将建立起完整实用的 DNS 知识体系，从容应对各种域名解析需求。
