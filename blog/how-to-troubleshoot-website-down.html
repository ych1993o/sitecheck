网站打不开？完整排查指南

2026年7月2日 · SiteCheck

"网站打不开了！"——这大概是运维和站长最怕听到的一句话。网站无法访问的原因五花八门：可能是 DNS 解析失败、服务器宕机、SSL 证书出问题，也可能是防火墙拦截或应用代码崩溃。面对故障，慌乱地到处试往往浪费时间，正确的做法是按层次系统地排查。本文提供一套从用户端到服务器端的完整排查流程，帮助你快速定位故障根因并恢复服务。

一、排查前的准备工作

开始排查前，先确认几个关键信息，避免走弯路：

  • 是所有人打不开，还是只有你打不开？用手机 4G 网络、问朋友、或用在线检测工具（如 SiteCheck）测试。如果别人能打开，问题多半在你的本地网络或设备。
  • 是间歇性打不开，还是完全打不开？间歇性通常是服务器过载或网络抖动，完全打不开更可能是 DNS 或服务宕机。
  • 具体是什么错误提示？"无法访问此网站""连接超时""SSL 错误""502 Bad Gateway"——不同提示指向不同问题层级。

带着这些信息，我们按从下到上的层次逐一排查。

二、第一层：DNS 解析排查

DNS 是访问的第一步，域名解析不成功，后面一切都是空谈。判断 DNS 是否正常的方法：

  1. ping 域名：ping example.com，看是否能解析出 IP。如果提示"找不到主机"，基本就是 DNS 问题。
  2. dig/nslookup：dig example.com 查看详细解析过程和返回的 A 记录。
  3. 在线 DNS 查询：用 DNS 查询工具从多个节点检测解析结果是否一致。

DNS 层常见问题：

  • 域名过期：忘了续费，域名被注册商暂停解析。检查域名注册商后台。
  • DNS 记录配置错误：A 记录指向了错误 IP，或 NS 记录没正确指向 DNS 服务商。用 dig 确认权威服务器返回的记录。
  • DNS 服务器故障：DNS 服务商自身宕机。换用公共 DNS（如 8.8.8.8、1.1.1.1）测试。
  • DNS 传播中：刚修改记录，部分地区还没生效。关于传播机制，参考DNS 传播原理。

如果 DNS 解析正常，能拿到正确 IP，进入下一层。

三、第二层：网络连通性排查

DNS 正常但打不开，可能是网络层不通。排查服务器 IP 是否可达：

  1. ping IP：ping 192.0.2.1，直接 ping 服务器 IP（绕过 DNS）。不通可能是服务器关机、防火墙禁 ICMP 或网络中断。
  2. traceroute：traceroute example.com，追踪数据包路径，看在哪一跳断掉，判断是本地网络、运营商还是机房问题。
  3. telnet/nc 测试端口：telnet example.com 443 或 nc -zv example.com 443，测试 80/443 端口是否通。端口不通通常是防火墙或安全组没放行。

网络层常见问题：

  • 服务器宕机：登录云控制台看实例状态，可能需要重启。
  • 安全组/防火墙：云服务器安全组没放行 80/443 端口，或系统防火墙（iptables/ufw/firewalld）拦截。
  • IP 被封：服务器 IP 因违规或被攻击被运营商封禁。
  • 本地网络问题：DNS 缓存中毒、代理软件干扰、hosts 文件错误配置。清 DNS 缓存：ipconfig /flushdns（Windows）或 systemd-resolve --flush-caches（Linux）。

四、第三层：SSL/TLS 排查

端口通了但浏览器报 SSL 错误，问题在证书层。常见 SSL 错误及原因：

  • NET::ERR_CERT_DATE_INVALID：证书过期。解决方案见SSL 证书过期解决方案。
  • NET::ERR_CERT_COMMON_NAME_INVALID：证书域名不匹配，访问的域名不在证书 SAN 列表。
  • NET::ERR_CERT_AUTHORITY_INVALID：证书链不完整或用了不受信任的 CA。
  • ERR_SSL_PROTOCOL_ERROR：协议版本不匹配，如服务器只支持 TLS 1.2 而客户端太旧。

用 SSL 证书检测工具或 openssl s_client -connect example.com:443 深入诊断证书状态。更多证书检测方法可参考如何检测 SSL 证书。

五、第四层：HTTP/Web 服务器排查

SSL 正常但返回错误状态码，问题在 Web 服务器或应用层。先看状态码：

  • 连接被拒（Connection refused）：Web 服务（Nginx/Apache）没启动或没监听对应端口。登录服务器 systemctl status nginx 检查。
  • 403 Forbidden：文件权限不对、目录无 index、或 Nginx 配置 deny。检查网站根目录权限和配置。
  • 404 Not Found：根目录为空或 root 路径配置错误。
  • 500 Internal Server Error：应用代码出错。看应用日志和 Nginx error.log 定位异常。
  • 502 Bad Gateway：反向代理后端服务挂了。检查 PHP-FPM、Node.js 等后端进程是否存活、端口是否监听。
  • 504 Gateway Timeout：后端响应超时。可能是数据库慢查询或后端处理不过来。

排查 Web 服务器问题，日志是最好的朋友。Nginx 的 access.log 和 error.log、应用的运行日志，往往直接给出错误原因。状态码的含义可参考HTTP 状态码详解。

六、第五层：应用与数据库排查

Web 服务器正常但应用异常，进入应用层排查：

  • 数据库连接失败：应用连不上数据库会报 500。检查数据库是否运行、连接数是否打满、账号密码是否正确、白名单是否放行应用服务器 IP。
  • 应用进程崩溃：OOM（内存不足）被系统杀进程，或代码异常退出。看 dmesg 和应用日志。
  • 磁盘满：日志或上传文件把磁盘写满，导致服务异常。df -h 查看磁盘，清理空间。
  • 依赖服务故障：Redis、Elasticsearch 等依赖服务挂掉，级联影响应用。
  • 配置错误：最近部署改了配置（环境变量、连接串）导致应用起不来。

七、常见故障场景速查

把上面的排查流程浓缩成几个典型场景的速查：

找不到主机/DNS 错误域名过期、DNS 记录错、DNS 服务商故障dig、DNS 查询工具

八、建立故障排查的标准化流程

故障排查不是临时抱佛脚，而应建立标准化流程，做到"闻警即动，按图索骥"：

  1. 分层定位：按 DNS → 网络 → SSL → Web 服务器 → 应用的顺序逐层排除，避免跳跃式排查。
  2. 最小化变量：每次只改一个地方，观察效果，便于定位因果。
  3. 留痕记录：排查过程记录时间线、操作和现象，事后复盘改进。
  4. 监控先行：用 网站监控在故障发生第一时间告警，而非等用户反馈。
  5. 应急预案：准备常见故障的应急手册（域名续费流程、证书续期步骤、服务器重启方法），缩短恢复时间。

九、善用工具提效

排查效率很大程度取决于工具熟练度。推荐常备以下工具：

  • 在线综合检测：SiteCheck 的 HTTP 状态、DNS 查询、SSL 检测、响应时间、HTTP Headers 工具，覆盖大部分排查需求。
  • 命令行：ping、dig、traceroute、curl、telnet、openssl，运维必备。
  • 日志分析：grep/journalctl 快速过滤日志，或用 ELK 等日志平台。
  • 监控告警：Prometheus + Grafana + AlertManager，故障秒级感知。

十、总结

网站打不开并不可怕，可怕的是没有章法地乱试。本文提供了一套从 DNS、网络、SSL、Web 服务器到应用层的分层排查流程，每个层级都给出了具体的检测方法和常见问题。核心思路是：先判断影响范围，再按层次逐层排除，用工具和数据说话，而不是凭猜测。

建议你把这套排查流程整理成自己的应急手册，遇到故障时按步骤执行。更重要的是，通过持续监控把故障发现提前到用户感知之前。排查能力是运维的基本功，多经历几次故障、多复盘总结，你的排查速度和准确度都会显著提升。当你能从容应对各种"打不开"的告警时，就已经是一名合格的网站守护者了。
