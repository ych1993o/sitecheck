Complete Guide: Migrating Your Website from HTTP to HTTPS

Published July 2026 · SSL/TLS · ~12 min read

In 2026, HTTPS is not optional. Google Chrome marks all HTTP websites as "Not Secure" in the address bar. Modern browsers restrict access to powerful APIs (geolocation, camera, microphone, service workers) to secure origins only. Google uses HTTPS as a ranking signal in search results. And users have learned to associate the padlock icon with trustworthiness. If your website still runs on HTTP, you are losing search traffic, user trust, and access to essential browser features.

Migrating from HTTP to HTTPS is a multi-step process that goes beyond simply installing an SSL certificate. Done incorrectly, you can break links, lose search engine rankings, create mixed content errors, and confuse your users. This comprehensive guide walks you through every step of the migration process, from choosing the right certificate type to configuring your server, fixing mixed content, setting up HSTS, and preserving your SEO rankings.

Why HTTPS Matters

HTTPS (Hypertext Transfer Protocol Secure) encrypts all communication between the user's browser and your web server using TLS (Transport Layer Security). This encryption serves three critical purposes.

Confidentiality: Encryption ensures that no one in the middle—whether an ISP, a coffee shop Wi-Fi operator, or a malicious actor on the same network—can read the data being transmitted. This protects login credentials, personal information, payment details, and any other sensitive data your users submit.

Integrity: TLS ensures that the data received by the browser is exactly what the server sent. Without encryption, an attacker on the network could modify the content in transit—injecting ads, replacing download links, or altering page content.

Authentication: An SSL certificate verifies that the server your users are connecting to is actually operated by the domain owner they expect. This prevents man-in-the-middle attacks where an attacker impersonates your website.

Beyond security, HTTPS is a practical necessity. Google has stated that HTTPS is a lightweight ranking signal, meaning secure sites have a slight advantage in search results. The Chrome browser displays a "Not Secure" warning for all HTTP pages, which can significantly increase bounce rates. Progressive Web Apps (PWAs), which can work offline and provide app-like experiences, require HTTPS. And many third-party services, including payment processors and social login providers, require their integration pages to be served over HTTPS.

Understanding SSL Certificate Types

SSL certificates (more accurately called TLS certificates) come in three main validation levels, each providing a different degree of identity verification.

Domain Validation (DV) certificates: DV certificates verify only that the certificate applicant controls the domain. Verification is typically automated—either by adding a DNS TXT record or placing a verification file on the web server. DV certificates can be issued in minutes and are free through Let's Encrypt. They provide encryption but do not verify the identity of the organization behind the website. For most websites, DV certificates are sufficient.

Organization Validation (OV) certificates: OV certificates include verification of the organization's legal identity, physical address, and domain ownership. The Certificate Authority performs manual checks to confirm the organization is legitimate. OV certificates display the organization's name in the certificate details (visible by clicking the padlock icon). They are typically used by businesses that want to provide additional trust signals to their customers. OV certificates generally cost $50 to $200 per year.

Extended Validation (EV) certificates: EV certificates require the most rigorous verification process, including detailed checks into the organization's legal, physical, and operational existence. Historically, EV certificates triggered a green address bar in browsers, providing a strong visual trust indicator. However, as of 2026, all major browsers have removed the green bar and now treat EV certificates similarly to OV certificates in the UI. EV certificates are expensive ($150 to $500+ per year) and are now primarily used by financial institutions and high-security environments where the additional verification is required by regulation.

Wildcard certificates: A wildcard certificate (available as DV, OV, or EV) secures a domain and all its subdomains with a single certificate—for example, *.example.com covers www.example.com, blog.example.com, api.example.com, and any other subdomain. This is convenient and cost-effective for sites with many subdomains but note that a wildcard only covers one level of subdomains.

Multi-domain (SAN) certificates: Subject Alternative Name certificates allow you to secure multiple different domains with a single certificate—for example, both example.com and example.org. This is useful for organizations that manage multiple related domains.

Getting a Free SSL Certificate with Let's Encrypt

Let's Encrypt is a free, automated, and open Certificate Authority that has made HTTPS accessible to every website owner. Certificates are valid for 90 days, and the renewal process is fully automatable using the Certbot tool.

Using Certbot (recommended): Certbot is a free tool developed by the Electronic Frontier Foundation that automates the entire process of obtaining and installing Let's Encrypt certificates. It supports Nginx, Apache, and various other web servers.

For Nginx, the process is straightforward:

# Install Certbot
sudo apt update
sudo apt install certbot python3-certbot-nginx

# Obtain and install a certificate
sudo certbot --nginx -d example.com -d www.example.com

# Certbot will automatically modify your Nginx configuration
# and set up automatic renewal via a systemd timer or cron job

For Apache:

# Install Certbot
sudo apt update
sudo apt install certbot python3-certbot-apache

# Obtain and install a certificate
sudo certbot --apache -d example.com -d www.example.com

Certbot automatically configures your web server, sets up redirects, and installs a cron job or systemd timer that automatically renews certificates before they expire. You can verify automatic renewal with sudo certbot renew --dry-run.

For wildcard certificates: Let's Encrypt supports wildcard certificates, but they require DNS-01 challenge validation. This means you must add a specific TXT record to your DNS configuration. Certbot supports this with the appropriate DNS plugin (e.g., certbot-dns-cloudflare for Cloudflare DNS).

Alternative clients: While Certbot is the most popular, alternatives include acme.sh (a lightweight shell script), Caddy (a web server with automatic HTTPS built in), and cloud provider integrations (AWS Certificate Manager, Cloudflare Origin CA). Each has its own advantages depending on your infrastructure.

Server Configuration for HTTPS

After obtaining your certificate, you need to configure your web server to use it. Here are the essential configurations for Nginx and Apache.

Nginx Configuration

Create or modify your Nginx server block to listen on port 443 with SSL:

server {
    listen 80;
    server_name example.com www.example.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name example.com www.example.com;

    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # Modern TLS configuration
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;

    # Session settings for performance
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;
    ssl_session_tickets off;

    # OCSP Stapling
    ssl_stapling on;
    ssl_stapling_verify on;
    resolver 8.8.8.8 8.8.4.4 valid=300s;

    root /var/www/example.com;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }
}

Key points: The first server block redirects all HTTP traffic to HTTPS. The second block serves content over HTTPS with modern TLS protocols (1.2 and 1.3), strong cipher suites, OCSP stapling for faster certificate validation, and session caching for better performance.

Apache Configuration

<VirtualHost *:80>
    ServerName example.com
    ServerAlias www.example.com
    Redirect permanent / https://example.com/
</VirtualHost>

<VirtualHost *:443>
    ServerName example.com
    ServerAlias www.example.com

    SSLEngine on
    SSLCertificateFile /etc/letsencrypt/live/example.com/cert.pem
    SSLCertificateKeyFile /etc/letsencrypt/live/example.com/privkey.pem
    SSLCertificateChainFile /etc/letsencrypt/live/example.com/chain.pem

    SSLProtocol all -SSLv3 -TLSv1 -TLSv1.1
    SSLCipherSuite ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384
    SSLHonorCipherOrder off

    Header always set Strict-Transport-Security "max-age=63072000; includeSubDomains; preload"

    DocumentRoot /var/www/example.com
</VirtualHost>

After modifying your configuration, test it with sudo nginx -t (Nginx) or sudo apachectl configtest (Apache), then reload the server.

Fixing Mixed Content

Mixed content occurs when an HTTPS page loads resources (images, scripts, stylesheets, iframes, fonts) over HTTP. Browsers block or warn about mixed content because it undermines the security of the HTTPS page—an HTTP resource could be modified by an attacker in transit.

Types of mixed content: Browsers categorize mixed content as either "passive" (images, audio, video) or "active" (scripts, stylesheets, iframes, XMLHttpRequests). Modern browsers block active mixed content entirely and may downgrade or block passive mixed content. Chrome plans to eventually block all mixed content.

How to find mixed content: Open your browser's Developer Tools (F12), go to the Console tab, and look for messages like "Mixed Content: The page at '...' was loaded over HTTPS, but requested an insecure resource '...'." You can also use the Security tab in Chrome DevTools to see a detailed list of mixed content resources.

Fixing strategies:

1. Use protocol-relative or HTTPS URLs: Change all http:// resource URLs in your HTML, CSS, and JavaScript to https://. Alternatively, use protocol-relative URLs (//example.com/image.png), which automatically use the parent page's protocol.

2. Update database content: If your CMS stores URLs with http:// in the database (common in WordPress content), run a search-and-replace query. For WordPress, tools like Better Search Replace or WP-CLI can handle this safely.

3. Upgrade Insecure Requests header: Add the Content-Security-Policy header with the upgrade-insecure-requests directive. This tells the browser to automatically upgrade all HTTP requests to HTTPS:

Content-Security-Policy: upgrade-insecure-requests

This is a quick fix that works for most cases, but you should still update your source code to use HTTPS URLs directly for clarity and reliability.

4. Update third-party resources: External scripts, fonts, and embedded content from third parties may use HTTP URLs. Check each third-party resource and update to their HTTPS version. Most reputable services now provide HTTPS endpoints. If a third-party resource does not support HTTPS, consider finding an alternative.

Configuring HSTS (HTTP Strict Transport Security)

HSTS is a security header that tells the browser to always connect to your site via HTTPS, even if the user types "http://" in the address bar. This prevents SSL stripping attacks, where an attacker on the network downgrades the connection to HTTP to intercept traffic.

Basic HSTS configuration: Add the following header to your HTTPS responses:

Strict-Transport-Security: max-age=31536000; includeSubDomains

This tells the browser to use HTTPS for all requests to this domain and its subdomains for one year (31,536,000 seconds). Once a browser receives this header, it will refuse to connect via HTTP for the specified duration, even if the user explicitly types an HTTP URL.

HSTS preload: You can submit your domain to the HSTS Preload List (hstspreload.org), which is built into all major browsers. Once preloaded, browsers will use HTTPS for your domain even on the very first visit, before receiving the HSTS header. This provides the strongest possible protection. Requirements for preload include: serve the HSTS header for at least one year, include subdomains, and have a valid SSL certificate for the base domain and all subdomains.

Warning: HSTS is a strong commitment. Once browsers have cached the HSTS policy, you cannot easily disable HTTPS without waiting for the max-age to expire. Start with a short max-age (e.g., 300 seconds) to test, then increase it gradually to ensure everything works correctly before setting a long duration or submitting to the preload list.

SEO Migration Considerations

Switching from HTTP to HTTPS changes your page URLs, which can impact your search engine rankings if not handled correctly. Follow these steps to preserve your SEO value during the migration.

Implement 301 redirects: Redirect every HTTP URL to its HTTPS counterpart using 301 (permanent) redirects. This tells search engines that the content has permanently moved, and they should transfer the ranking signals (link equity, crawl signals) from the HTTP URL to the HTTPS URL. Ensure redirects are done at the server level (Nginx/Apache), not with JavaScript or meta refreshes.

Update internal links: Update all internal links in your site to point to HTTPS URLs directly. While 301 redirects will handle the transition, direct HTTPS links are more efficient and provide a cleaner user experience.

Update sitemaps: Generate new XML sitemaps with all URLs using HTTPS. Submit the updated sitemaps to Google Search Console and Bing Webmaster Tools.

Update Google Search Console: Add your HTTPS site as a new property in Google Search Console. Submit the updated sitemap. Monitor the coverage report to ensure Google is crawling and indexing your HTTPS pages. Keep both the HTTP and HTTPS properties verified during the transition period.

Update canonical tags: Ensure that canonical tags on your pages point to the HTTPS version. If you use rel="canonical" to indicate the preferred URL, it should reference the HTTPS URL.

Update external references: Where possible, update backlinks from external sites, social media profiles, Google Business Profile, and any other external references to use HTTPS URLs. While 301 redirects will pass link equity, direct links are always preferable.

Use the Change of Address tool: In Google Search Console, use the Change of Address tool (under Settings) to inform Google that your site has moved from HTTP to HTTPS. This helps accelerate the transfer of ranking signals.

Performance Impact Analysis

A common concern about HTTPS migration is that the additional encryption overhead will slow down the website. In practice, the performance impact of TLS in 2026 is minimal and often negligible.

TLS handshake overhead: The initial TLS handshake adds one to two additional round trips before the first HTTP request can be sent. With TLS 1.3, the handshake is reduced to one round trip (or zero with session resumption and 0-RTT). For users on modern networks, this adds roughly 50-100ms on the first connection. Subsequent connections to the same server reuse the TLS session and incur no additional overhead.

Encryption overhead: The actual encryption and decryption of data is extremely fast on modern hardware. AES-NI instruction sets, available on virtually all processors manufactured in the last decade, make AES encryption nearly as fast as unencrypted data transfer. The performance cost of encryption is typically less than 1% of total server CPU usage for most websites.

Mitigating TLS overhead: Use TLS 1.3, which is faster than TLS 1.2. Enable OCSP stapling to avoid additional lookups for certificate revocation status. Enable session caching and session tickets for faster resumed connections. Serve your site through a CDN, which can handle TLS termination at the edge and use optimized, persistent connections to your origin server.

HTTPS can actually improve performance: With HTTP/2 and HTTP/3 requiring HTTPS, enabling TLS unlocks these faster protocols. HTTP/2's multiplexing, header compression, and server push can significantly improve page load times compared to HTTP/1.1. The net performance impact of HTTPS with HTTP/2 is often positive compared to unencrypted HTTP/1.1.

Post-Migration Verification Checklist

After completing your HTTPS migration, verify the following items to ensure everything is configured correctly:

  • All pages load correctly over HTTPS with no errors
  • HTTP URLs redirect to HTTPS with 301 status codes
  • No mixed content warnings in the browser console
  • The SSL certificate is valid and correctly installed (check with SSL Labs)
  • Security headers are present (HSTS, CSP, X-Frame-Options, etc.)
  • All internal links point to HTTPS URLs
  • The XML sitemap uses HTTPS URLs
  • Google Search Console shows the HTTPS property is being indexed
  • Analytics tracking is working on the HTTPS site
  • Third-party integrations (payments, social logins, embedded content) work over HTTPS

Conclusion

Migrating from HTTP to HTTPS is a critical step for any website in 2026. The process involves obtaining an SSL certificate, configuring your web server, fixing mixed content, implementing proper redirects, updating your SEO configuration, and setting up HSTS for long-term security. While the migration requires careful planning and execution to avoid breaking things, the tools available today—especially free certificates from Let's Encrypt and automated tools like Certbot—make the process more accessible than ever. The benefits are substantial: improved security, better search rankings, enhanced user trust, and access to modern web features. Plan your migration carefully, follow the steps outlined in this guide, and your website will be securely serving visitors over HTTPS with minimal disruption.
