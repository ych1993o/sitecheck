HTTP响应头详解与安全配置指南

2026年7月4日 · SiteCheck

每次浏览器向服务器发送请求，服务器返回的不仅是状态码和内容，还有一组 HTTP 响应头（Response Headers）。这些键值对承载着缓存控制、内容类型、安全策略、跨域规则等关键信息，是浏览器与服务器"对话"的重要语言。理解响应头，才能精确控制网站的行为和安全。本文将详解常见响应头的作用，并给出安全配置实践。

一、HTTP 头的基础概念

HTTP 头分为请求头（Request Headers）和响应头（Response Headers）。请求头由浏览器发送，告诉服务器客户端的信息和需求；响应头由服务器返回，告诉浏览器如何处理响应。两者都是"键: 值"格式，每行一个。

响应头又可细分为：通用头（如 Date、Connection）、实体头（描述响应体，如 Content-Type、Content-Length）和扩展头。掌握常用响应头，能帮助你诊断缓存问题、排查跨域错误、加固网站安全。

想查看一个网站的实际响应头，最方便的方式是用 SiteCheck 的 HTTP Headers 检测工具，或用 curl -I https://example.com 命令。

二、内容与类型相关响应头

Content-Type

告诉浏览器响应体的 MIME 类型，决定了浏览器如何解析内容。例如 text/html; charset=utf-8 表示 HTML 文档，application/json 表示 JSON 数据。如果 Content-Type 设置错误，浏览器可能把 HTML 当纯文本显示，或导致乱码。务必声明正确的 charset，避免编码问题。

Content-Length

响应体的字节长度，让浏览器知道要接收多少数据，便于显示进度条和判断是否传输完整。

Content-Encoding

响应体的压缩编码，如 gzip、br（Brotli）。浏览器据此解压内容。配合压缩能大幅减少传输量。

Content-Disposition

控制浏览器如何处理响应体。inline 表示直接显示，attachment; filename="report.pdf" 表示下载并指定文件名。文件下载场景必用。

三、缓存控制响应头

缓存是性能优化的核心，而缓存行为由响应头控制。理解这几个头至关重要。

Cache-Control

最强大的缓存控制头，常用指令包括：

  • max-age=31536000：缓存有效期 1 年（秒）。
  • public：可被浏览器和 CDN 等中间代理缓存。
  • private：只允许浏览器缓存，CDN 不缓存（适用于用户个性化内容）。
  • no-cache：每次使用前必须向服务器验证（配合 ETag）。
  • no-store：完全不缓存，适用于敏感数据。
  • immutable：资源永不变，无需验证（用于带哈希的文件名）。

ETag 与 Last-Modified

这两个头用于协商缓存。ETag 是资源的唯一标识（通常是内容哈希），Last-Modified 是最后修改时间。浏览器下次请求会带上 If-None-Match 或 If-Modified-Since，服务器比对后若未变化则返回 304，让浏览器用缓存。ETag 比 Last-Modified 更精确，优先级也更高。

Expires

老式缓存头，指定一个绝对过期时间（如 Expires: Wed, 09 Jul 2026 00:00:00 GMT）。已被 Cache-Control 取代，但为兼容旧客户端仍可保留。当 Cache-Control 和 Expires 同时存在时，Cache-Control 优先。

四、跨域与 Cookie 相关响应头

Access-Control-Allow-Origin

CORS（跨域资源共享）的核心头。浏览器有同源策略，默认禁止跨域请求。服务器通过这个头声明允许哪些源访问资源。例如 Access-Control-Allow-Origin: https://example.com 只允许该源跨域访问，* 表示允许任意源（但不适用于带凭证的请求）。

Access-Control-Allow-Methods / Headers

预检请求（OPTIONS）的响应中，声明允许的 HTTP 方法和自定义头。如 Access-Control-Allow-Methods: GET, POST, PUT, DELETE。

Set-Cookie

服务器通过这个头向浏览器设置 Cookie。一个响应可包含多个 Set-Cookie。关键属性包括：

  • HttpOnly：禁止 JS 访问 Cookie，防 XSS 窃取。
  • Secure：只在 HTTPS 下发送。
  • SameSite=Lax|Strict|None：控制跨站请求是否携带 Cookie，防 CSRF。
  • Max-Age / Expires：Cookie 生命周期。

五、安全相关响应头

安全头是抵御 Web 攻击的第一道防线。这里做概览，详细的配置方法请阅读HTTP 安全头配置指南。

Strict-Transport-Security (HSTS)

强制浏览器后续访问一律使用 HTTPS，防止降级攻击和 SSL 剥离。典型配置：Strict-Transport-Security: max-age=31536000; includeSubDomains; preload。

Content-Security-Policy (CSP)

限制页面能加载哪些资源，是防御 XSS 的利器。例如 default-src 'self' 只允许加载同源资源。CSP 配置需要逐步收紧，避免误伤正常功能。

X-Frame-Options

控制页面是否允许被 iframe 嵌入，防点击劫持。DENY 完全禁止，SAMEORIGIN 只允许同源嵌入。新标准用 CSP 的 frame-ancestors 替代。

X-Content-Type-Options

设为 nosniff，禁止浏览器 MIME 嗅探，防止把非脚本文件当脚本执行。

Referrer-Policy

控制 Referer 头的发送策略，如 strict-origin-when-cross-origin，保护用户隐私不泄露完整 URL。

六、其他实用响应头

Location

配合 3xx 重定向状态码使用，指向新的 URL。也用于 201 Created 指向新创建资源的地址。

Server

服务器软件标识，如 nginx/1.24.0。出于安全考虑，建议隐藏或修改此头，避免暴露服务器版本给攻击者提供信息。

X-Request-ID / X-Correlation-ID

请求追踪标识，在微服务架构中用于串联一次请求经过的所有服务，便于日志排查。这是自定义头的常见实践。

Vary

告诉缓存服务器，响应内容会随哪些请求头变化。例如 Vary: Accept-Encoding 表示对支持 gzip 和不支持的客户端要分别缓存。配置不当会导致缓存串内容，需谨慎。

七、Nginx 配置响应头实战

在 Nginx 中添加响应头非常简单，用 add_header 指令：

server {
    listen 443 ssl http2;
    server_name example.com;

    # 安全头
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # 静态资源长缓存
    location ~* \.(js|css|png|jpg|webp)$ {
        add_header Cache-Control "public, max-age=31536000, immutable";
    }

    # HTML 协商缓存
    location / {
        add_header Cache-Control "no-cache";
    }
}

注意 add_header 在 location 块中会覆盖上层继承的头，所以安全头需要在每个 location 重复声明，或使用 include 复用。加 always 参数确保错误响应也带上安全头。

八、响应头常见问题排查

  • 资源不缓存，每次都重新下载：检查 Cache-Control 是否被设成 no-store，或被 no-cache 误用。
  • 跨域请求被浏览器拦截：看响应是否带正确的 Access-Control-Allow-Origin，且与请求 Origin 匹配。
  • 页面被恶意 iframe 嵌套：配置 X-Frame-Options 或 CSP frame-ancestors。
  • Cookie 被 XSS 偷走：确认 Cookie 设置了 HttpOnly 和 Secure。
  • 缓存串内容（移动端看到桌面版）：检查 Vary 头是否包含 User-Agent 或 Accept。

九、如何检测自己的响应头

配置完成后务必验证。推荐方式：

  • 在线工具：SiteCheck 的 HTTP Headers 检测工具，可视化展示所有头并标注安全头缺失情况。
  • curl：curl -I https://example.com 快速查看。
  • 浏览器开发者工具：Network 面板点击请求，查看 Response Headers。
  • 安全头评分：用 securityheaders.com 等工具给安全头配置打分，查漏补缺。

十、总结

HTTP 响应头是控制网站行为和安全的关键手段。本文系统讲解了内容类型、缓存控制、跨域、Cookie、安全策略等几大类响应头的作用与配置。核心要点：用 Cache-Control 和 ETag 精确控制缓存；用 CORS 头解决跨域；用 Set-Cookie 的 HttpOnly/Secure/SameSite 保护会话；用 HSTS、CSP、X-Frame-Options 等安全头构筑防线。

响应头配置虽然简单，但影响深远。建议你用检测工具审视一下自己网站的响应头，补齐缺失的安全头，优化缓存策略。一个配置良好的响应头集合，能让网站更快、更安全、更可控。结合本文和后续的安全头专题文章，你将建立起完整的 HTTP 头知识体系。
